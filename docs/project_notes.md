# Opir Guardrail Models

As LLMs become more capable of autonomous actions, safeguard middleware between an agent and the user becomes increasingly important. Existing safeguard models are often large autoregressive models with 7B-22B parameters, which makes large-scale deployment expensive. Many of these systems are also fine-tuned mainly for binary safe/unsafe classification, while true zero-shot categorization across safety domains remains limited.

This project introduces a family of encoder-based guardrail models built on the GLiClass architecture. The released models include multi-task models for safe/unsafe classification, toxicity classification, jailbreak classification, and zero-shot unsafe prompt and response categorization. The project includes both English-only and multilingual models, as well as edge models with fewer than 100M parameters for safe/unsafe categorization.

## Model Families

- `Opir-multitask-large`: based on DeBERTaV3-large and fine-tuned from `knowledgator/gliclass-instruct-large-v1.0`.
- `Opir-multitask-x`: based on mDeBERTaV3-base and fine-tuned from `knowledgator/gliclass-x-base`.
- `Opir-edge`: based on `knowledgator/gliclass-edge-v3.0` with the `jhu-clsp/ettin-encoder-32m` backbone.
- `Opir-edge-x`: based on `knowledgator/gliclass-multilang-edge` with the `jhu-clsp/mmBERT-small` backbone.

## Safety Taxonomy

The dataset was built around a three-level safety taxonomy designed to cover diverse safety risks:

- Level 1 labels: 16
- Level 2 labels: 126
- Level 3 labels: 854
- Total labels across all levels: 996

First-level labels:

- `toxicity`
- `violence_and_physical_harm`
- `self_harm_and_suicide`
- `sexual_content`
- `child_safety`
- `personal_information_privacy_and_intellectual_property`
- `cybersecurity`
- `criminal_and_illegal_activity`
- `regulated_goods_and_advice`
- `biological_medical_and_environmental_harm`
- `weapons_of_mass_destruction`

- `other_or_uncertain`
- `safe_and_benign`

## Dataset Construction

For each node in the taxonomy, 30 unsafe prompts were generated. Hard negatives were then mined by evolutionarily modifying initial prompts to bypass existing safety models. LLMs were used as judges to validate whether a prompt remained unsafe, using a mixture of DeepSeek-V3.1, MiniMax-M2.5, and Meta-Llama-3.3-70B-Instruct.

Hard-positive prompts were also generated by prompting LLMs to produce benign or safety-preserving examples that can be difficult for classifiers because they contain safety-related language without requiring an unsafe label.

To obtain diverse response examples, a Qwen3-4B model was fine-tuned on Aegis2 and used to generate responses for the prompts created in the first stage. Responses were then classified for safety with an LLM-as-a-judge pipeline.

The multilingual dataset was created by translating prompts and responses with DeepSeek-V3.1 into Swedish, Norwegian, Czech, Polish, Lithuanian, Estonian, Latvian, Spanish, Finnish, English, German, French, Romanian, Italian, Portuguese, Dutch, Ukrainian, Russian, Hindi, Chinese, Japanese, Korean, and Arabic.

The local dataset files include:

- `data/gliclass_safety_multi.json`: 531,007 multilingual safety examples.
- `data/gliclass_safety_en.json`: 213,809 English safety examples.
- `data/gliclass_full_multi.json`: 1,106,635 multilingual multi-task examples.
- `data/gliclass_full_en.json`: 426,356 English multi-task examples.
- `data/gliclass_post_training.json`: 18,000 post-training examples.

The training data also includes portions of the Aegis2 and WildGuard training subsets.

## Training Setup

Training is implemented in `train.py`. The script loads the JSON training dataset, extends it with `knowledgator/gliclass-v3-logic-dataset`, shuffles the combined data, and uses a 90/10 train/evaluation split.

Training was performed in two stages:

1. Initial training on the main dataset for 3 epochs.
2. Post-training on a 10% sample of examples after augmentation.

Important training hyperparameters used by the training script:

- Problem type: `multi_label_classification`
- Architecture type: `uni-encoder`
- Pooling: average pooling
- Class-token pooling: first token
- Max sequence length: 1024
- Batch size: 8
- Gradient accumulation steps: 1
- Encoder learning rate: `1e-6`
- Other/head learning rate: `3e-6`
- Encoder weight decay: `0.01`
- Other/head weight decay: `0.01`
- Scheduler: cosine
- Warmup ratio: `0.05`
- Dropout: `0.3`
- Label shuffling: enabled
- Precision: `bf16=True`; `fp16=False` by default
- Checkpoint save interval: 1000 steps
- Checkpoint limit: 3
- Focal loss alpha: `0.7`
- Focal loss gamma: `-1`
- Focal loss reduction: `none`
- Contrastive loss coefficient: `0.0`

The training code also supports optional online Elastic Weight Consolidation for continual learning, with default `ewc_lambda=100.0`, `ewc_gamma=0.95`, and Fisher normalization enabled.

## Augmentation

Online training augmentation is implemented in `gliclass/data_processing.py` through `AugmentationConfig` and `DataAugmenter`. When enabled, each training item can be modified probabilistically:

- Random label removal: samples a subset of candidate labels and removes true labels no longer present in that subset.
- Random label addition: adds random labels from the dataset label pool to the candidate label list.
- Random text addition: prepends or appends another example's text when the two examples have no overlapping candidate labels.
- Synonym replacement: replaces labels with label synonyms when label-description metadata is available.
- Description insertion: prepends or appends short label descriptions to the text when metadata is available.
- Few-shot example insertion: attaches one or two example texts with overlapping labels.

The post-training stage uses a 10% sample of the main examples and applies these augmentations to improve robustness to label-set changes, prompt formatting variation, and few-shot contexts.

Additional prompt-injection augmentation is implemented in `data/create_unsafe_prompt_injection_dataset.py`. It samples rows from `data/gliclass_safety_en.json`, generates safe-looking distractor instructions such as label overrides and repeated `safe` tokens, and inserts them at random valid positions in the original text. The augmented rows preserve the original text, insertion offset, injection metadata, and source metadata.

## Evaluation

Evaluation is implemented in `test_guardrails.py`. The evaluator supports GLiClass and GLiNER2 backends and runs zero-shot classification with a configurable threshold, defaulting to `0.5`.

For multi-label categorization, predictions and labels are binarized with `MultiLabelBinarizer`, and the evaluator reports micro, macro, and weighted F1. For binary safety datasets, predicted and gold labels are normalized into `safe` and `unsafe`; non-safety labels such as `refusal` and `compliance` are skipped when they cannot be converted to a safety decision. Binary evaluation reports accuracy, micro F1, macro F1, weighted F1, per-label precision/recall/F1/support, and predicted/gold label counts.

The evaluation suite includes:

- OpenAI moderation-style safety and category evaluation
- Aegis prompt safety, response safety, and safety categories
- SimpleSafetyTests
- HarmBench prompts and responses
- PKU-SafeRLHF prompt/response safety
- BeaverTails
- XSTest
- WildGuardMix prompt safety, response safety, response refusal, and prompt subcategory
- PolyGuard prompt safety, response safety, response refusal, and prompt subcategory

Processed local evaluation datasets are stored under `data/eval` and `data/processed`. The processed GLiClass full dataset contains separate task views for safety, toxicity, jailbreak detection, and main-category classification.
